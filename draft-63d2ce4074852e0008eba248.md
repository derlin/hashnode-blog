Want a challenge? Try to understand how dev.to generates anchors, and to reproduce the algorithm using only simple regexes (no character classes allowed!) and built-in features of your favorite language (no libraries). I did. I wish I hadn't 🤪.

This article summarizes my findings, and how I came up with [this](https://github.com/derlin/bitdowntoc/blob/main/src/commonMain/kotlin/ch/derlin/bitdowntoc/AnchorGenerator.kt) code. Try it yourself → [bitdowntoc](https://derlin.github.io/bitdowntoc) (use the *dev.to* profile).

# Context

[bitdowntoc](https://derlin.github.io/bitdowntoc) is a Markdown TOC - Table of Contents - generator I have been developing for a while now. The logic is written in Kotlin common, with JS and JVM modules to offer both a web and a command-line interface. I lately introduced a new profile to support dev.to, which forced me to dig into dev.to anchors generation. I wasn't expecting so many challenges!

# A brief introduction to Kotlin Multiplatform

From the documentation:

> The Kotlin Multiplatform technology is designed to simplify the development of cross-platform projects. It reduces time spent writing and maintaining the same code for [different platforms](https://kotlinlang.org/docs/multiplatform.html#kotlin-multiplatform-use-cases) while retaining the flexibility and benefits of native programming.

In short, [Kotlin Multiplatform](https://kotlinlang.org/docs/multiplatform.html) makes it easy to reuse common code between different platforms such as mobile, web, and desktop. How? Thanks to Kotlin Common and specific compilers/transpilers!

**Kotlin Common** is a set of libraries and tools that allow developers to write common code for multiple platforms. To ensure interoperability, this common code is written using only a subset of the Kotlin language, called the **Kotlin Multiplatform Language (Kotlin MPL)**, which provides a limited set of APIs and constructs that are compatible with/compilable for multiple platforms. In other words, it is written in a way that is independent of the underlying runtime. As a consequence, it can be imported by any platform-specific code (Kotlin JVM, [Kotlin JS](https://kotlinlang.org/docs/js-overview.html), [Kotlin Android](https://kotlinlang.org/docs/android-overview.html), [Kotlin Native](https://kotlinlang.org/docs/native-overview.html), etc.).

In bitdowntoc, I use two platforms: JS and JVM. I write my shared logic (parsing the markdown, generating the TOC, inserting the anchors, etc.) in a common module. The I/O itself is platform-specific, so I work only with `String` (as input and output).

My JVM module takes care of parsing command-line arguments (using the amazing Kotlin library [clikt](https://ajalt.github.io/clikt/)), reading a markdown file, calling the common module and writing the output back to a file. My JS module creates the web interface (mix of HTML/CSS/Kotlin JS), handles options through a form, and reads/writes the markdown string from/to input fields.

![](https://cdn.hashnode.com/res/hashnode/image/upload/v1674902355196/30561b77-da77-498f-ace3-cc466af43293.png align="center")

At build time, Kotlin compiles the MPL along with the platform-specific code into a platform-specific binary format: a JAR for the JVM, and a set of HTML/CSS/JS files for the web. Note that for JS, Kotlin uses NodeJS and yarn to bundle both code and resources, so that you end up with a folder you can statistically deploy (try it, it is awesome!) - I deploy this folder in GitHub Pages using GitHub Actions.

# Some of the challenges of Kotlin common

## Regexes

To parse the markdown and generate the anchors/anchors links, I abuse regexes - **reg**ular **ex**pressions. Every language provides some support for regexes, often with slight differences in the engine - the way they are interpreted. In Java, for example, regexes are represented by the `Pattern` class, while JS uses the `RegExp` class. Both have completely different implementations, support and behaviors.

My logic however is not written in JS or Java: it uses Kotlin MPL. As explained above, Kotlin Common aims at interoperability and platform agnosticism. To support regexes, it thus provides a wrapper called `Regex`. Here is the *full* documentation:

> ##### **For JVM**: represents a compiled regular expression. Provides functions to match strings in text with a pattern, replace the found occurrences and split text around matches. For pattern syntax reference see [Pattern](https://docs.oracle.com/javase/8/docs/api/java/util/regex/Pattern.html).
> 
> **For JS**: represents a compiled regular expression. Provides functions to match strings in text with a pattern, replace the found occurrences and split text around matches. For pattern syntax reference see [MDN RegExp](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/RegExp#Special_characters_meaning_in_regular_expressions) and [http://www.w3schools.com/jsref/jsref\_obj\_regexp.asp](http://www.w3schools.com/jsref/jsref_obj_regexp.asp).
> 
> Note that `RegExp` objects under the hood are constructed with [the "u" flag](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/RegExp/unicode) that enables Unicode-related features in regular expressions. This also makes the pattern syntax more strict, for example, prohibiting unnecessary escape sequences.

In other words, the regex string that is passed as a constructor in common's class `Regex`, say `.*`, will be "copy-pasted" inside a `Pattern` in java - `Pattern.compile(".*")` - and a `RegExp` object in javascript with the unicode flag - `/.*/u`. Do you see the problem here?

It means any regex that you use in common code must:

1. be supported by both the Java and JS implementations, and
    
2. be interpreted the same by both engines.
    

Java supports the `\p{Latin}` character range, while javascript supports the `\p{Emoji}`. Java requires all special characters to be escaped with double backslashes, JS requires only one backslash. The more complex the regex, the higher the chance corner cases will be handled differently.

In other words? You are limited to very simple regexes with no character range (`\p{}`), and you need to test everything in both JS and Java very thoroughly.

## Limited libraries

Since we are working in a "common" context, we are limited to libraries that are also supporting common. This set is currently very limited! Whatever cool library you are used to in Java or Kotlin is likely only available for the JVM. No sanitization, emoji-removal libraries at hand, most operations must rely on "basic Kotlin" only!

# Let's talk about dev.to

## What exactly are anchors

Let's say you add the following heading to your article markdown:

```plaintext
## Hello dev.to!
```

When you preview or save it, it becomes:

```xml
<h2>
  <a name="hello-devto" href="#hello-devto"></a>
  Hello dev.to!
</h2>
```

As you can see, dev.to automatically adds a `<a>` with a `name` attribute. The latter is called an *anchor*: the `hello-devto` can be used in a *fragment link* for navigation, using hashtag + anchor: `#hello-devto` (see [Finally a clean and easy way to add a table of contents to dev.to articles 🤩](https://blog.derlin.ch/add-a-table-of-contents-to-devto-articles) for more information).

To generate a table of contents for dev.to, you need to know how those anchors are generated, to use the right fragment link.

## The mess of dev.to anchors

Since dev.to is based on forem, which is open-source, it wasn't difficult to find the exact code used to generate anchors. Here it is ([html\_rouge.rb](https://github.com/forem/forem/blob/main/app/lib/redcarpet/render/html_rouge.rb)):

```ruby
# .. other pre-processing ...

def slugify(string) # here, string = heading title (pre-processed)
	stripped = ActionView::Base.full_sanitizer.sanitize string
	stripped.downcase
     .gsub(EmojiRegex::RGIEmoji, "").strip
     .gsub(/[[:punct:]]/u, "")
     .gsub(/\s+/, "-")
end
```

This code goes through the following steps (simplified):

1. it *sanitizes* the title using a forem function. Sanitizing means stripping HTML tags, to make the content safe;
    
2. it lowercases the whole string (`.downcase`);
    
3. It strips the emojis using a Ruby library called [emoji\_regex](https://www.rubydoc.info/gems/emoji_regex/2.0.0), which is based upon a javascript library of the same name;
    
4. It removes the punctuations with a regex using a POSIX [character class](https://en.wikipedia.org/wiki/Regular_expression#Character_classes);
    
5. It replaces (consecutive) spaces with dashes.
    

It seems so straightforward, right? But as easy as it looks, it yields some very strange results which are difficult to re-implement in another language. Well, it yields strange results, END. Don't believe me? Have a look at the following markdown document, and try to come up with a valid Table of Content for dev.to:

```markdown

# Kotlin is `fun`
# `<hello href="http://link.me">` <hello> `&%£`
```

# Let's implement it in Kotlin common

## Step 1: sanitization

Sanitization usually means stripping HTML tags. It goes a bit further with dev.to anchors.

We can use the basic regex `<.*?>` to strip all HTML tags. The `?` makes the match *non-greedy*, meaning `<hello> <world>` will yield `<hello>` and `<world>` instead of `<hello> <world>`:

```kotlin
title.replace(Regex("<.*?>"), "")

// <hello> world -> world
// hello <a href="..." target="...">x</a> world <img/> -> hello world
```

Looks good. But what about this:

```markdown
# <foo attr="x"> 
# & <
# `<hello>`
```

In this case, the first heading `<foo ...>` will be stripped all right. The second heading doesn't contain HTML code. It does, however, contain HTML entities, which dev.to escapes. In other words, `&` becomes `&amp;`, `<` become `&lt;`, etc. HTML escaping is usual pre-processing stuff when generating HTML from markdown. So far, it makes sense. The third heading, however, behaves differently due to the backticks. For some reason, dev.to:

1. replaces the backticks with *raw* markers, thus becoming `{raw}<hello>{endrow}`;
    
2. escapes whatever HTML entity is inside the *raw markers*, instead of stripping the tags.
    

`` `<hello>` `` is thus transformed into `{raw}&lt;hello&gt;{endraw}`, which will be passed further down the pipeline, to end up `-raw-lt-hello-gt-endraw-` (the leading and trailing dashes are important): each backticks pair is translated into a raw/endraw pair. Well, usually. Imagine you make a mistake somewhere in your article and forget to close an inline code block:

```plaintext
`unclosed
# `hello`   => -endraw-hello-raw-
```

The `raw/endraw` are now reversed on all headings below! Let me tell you I didn't go that far, it is too convoluted. In other words, please use proper markdown if you plan to use bitdowntoc 🙃.

Finally, I didn't talk about markdown links. What happens when you put a markdown link in your heading?

```plaintext
# [check it out](https://example.com)
```

I first thought the link would be converted to HTML, and then processed. It doesn't. dev.to works on the markdown directly by "stripping" the link (the part in parenthesis), and keeping the display text. Our example thus becomes `check-it-out`. <sub>Funny note: if in you TOC you simply copy the heading as-is, everything will be messed up. Why? Because you would end up with a link (the one in the heading) inside another link (the TOC one). So you also need to strip the link in the title, not only the anchor, but it is another subject.</sub>

Knowing exactly what is going on, we can now implement it in Kotlin common using simple regexes and a proper ordering of method calls:

```kotlin
title
  .lowercase()
  // HTML in `` are escaped (we thus end with extra "lt" and "gt" in the anchor), but not removed completely
  .transformInlineCode { it.escapeHtmlEntities() }
  // other HTML tags <*> are removed
  .stripHtmlTags()
  // for markdown links [a](b), only the text "a" part is kept
  .stripMarkdownLinks()
  // devto transforms <, >, & into &lt; &gt; and &amp; before
  // stripping punctuation -> lt, gt, amp
  .escapeHtmlEntities()
  // for this one, keep reading!
  .replace(removedEmojisRegex, "")
  // weirdly enough, the trim is done before punctuation / backticks
  .trim()
  // backticks become "-raw-" / "-endraw-"
  // beware though: if an unclosed backtick is found before the title,
  // it messes up the order!
  // let's assume people write good markdown...
  .replaceBackTicksWithRawMarker()
```

Notice the order: the `trim` for example needs to be called *before* handling backticks (and punctuation), so we can preserve leading/trailing dashes in some cases (e.g. `raw`/`endraw` markers). We also need to escape HTML entities in backticks *before* stripping HTML tags, or they will disappear. Long story short, "*\[In nature\] everything is connected, everything is interwoven, everything changes with everything, everything merges from one into another*" (Gotthold Ephraim 😉).

Of course, most of those functions do not exist in Kotlin, it would be too easy, so let's implement them as extension functions to the `String` class:

```kotlin
fun String.escapeHtmlEntities() = this
  // don't use "&;" since this function can be called twice!
  // we will strip them later anyway
  .replace("&", "amp")
  .replace("<", "lt")
  .replace(">", "gt")

// apply some transformation to the content in backticks
fun String.transformInlineCode(transform: (String) -> String) =
  Regex("`(.*?)`").findAll(this).fold(this) { acc, match ->
      val insideText = match.groupValues[1]
      acc.replace(match.value, "`" + transform(insideText) + "`")
  }

fun String.stripHtmlTags() =
    replace(Regex("<.*?>"), "")

fun String.stripMarkdownLinks() =
  replace(Regex("\\[([^\\]]*)\\]\\([^)]*\\)"), "$1")

fun String.replaceBackTicksWithRawMarker(): String {
  var num = 0 // we need a state here
  return map {
    if (it != '`') it else {
        // don't use "-" since they are stripped later
        if (num++ % 2 == 0) " raw " else " endraw "
    }
  }.joinToString("")
}
```

## Step 2: Emojis

This part was the hardest. Contrary to other platforms, dev.to doesn't treat emojis like any other special character (that is, just stripping them all). It does specific processing, using a common rust library. The [library documentation](https://www.rubydoc.info/gems/emoji_regex/3.2.1) says:

> * `EmojiRegex::RGIEmoji` is the regex you most likely want. It matches all emoji recommended for general interchange, as defined by [the Unicode standard's `RGI_Emoji` property](https://unicode.org/reports/tr51/#def_rgi_set).
>     

Looking at the Unicode standard:

> [***ED-27***](https://unicode.org/reports/tr51/#def_rgi_set)***. RGI emoji set*** — The set of all emoji (characters and sequences) covered by [***ED-20***](https://unicode.org/reports/tr51/#def_basic_emoji_set), [***ED-21***](https://unicode.org/reports/tr51/#def_std_emoji_keycap_sequence_set), [***ED-22***](https://unicode.org/reports/tr51/#def_std_emoji_modifier_sequence_set), [***ED-23***](https://unicode.org/reports/tr51/#def_std_emoji_flag_sequence_set), [***ED-24***](https://unicode.org/reports/tr51/#def_std_emoji_tag_sequence_set), and [***ED-25***](https://unicode.org/reports/tr51/#def_emoji_ZWJ_sequences).
> 
> * This is the subset of all valid emoji (characters and sequences) recommended for general interchange.
>     
> * This corresponds to the **RGI\_Emoji** property.
>     

Follow the links if you are interested. I was personally overwhelmed by the descriptions, files, and exceptions, and couldn't find a simple list of all emojis falling into those categories. I tried many different things (remember that I cannot use regex character classes, even though one exists in Javascript - `\p{Emoji}` !), in vain.

The epiphany came after a long day of trial and error:

1. let's find a list of all emojis - ideally in the form of a regex (I used the one from [https://github.com/sweaver2112/Regex-combined-emojis](https://github.com/sweaver2112/Regex-combined-emojis))
    
2. paste it into a dev.to heading,
    
3. hit preview and see which emojis are still present in the generated anchor,
    
4. remove the emojis still in the anchor (3) from the regex (1).
    

Jackpot! The final regex is a bit long though, **12,364 characters** 🤪, but it does the job:

```kotlin
private val removedEmojisRegex = Regex(
  "🧑🏻❤️💋🧑🏼|🧑🏻❤️💋🧑🏽|...|🥭|🍎|🍏|...|Ⓜ|⚫|⚪|⬛|⬜|◾|◽"
)

title
  // ... sanitize ...
  .replace(removedEmojisRegex, "")
```

The regex even takes into account the emoji modifiers. The first two blocks (separated by `|`), actually represent the emojis 🧑🏻‍❤️‍💋‍🧑🏼 and 🧑🏻‍❤️‍💋‍🧑🏽 (kiss with different skin tones).

## Step 3: punctuations and spaces

The missing pieces are to strip punctuations and to concatenate consecutive spaces.

The original ruby code uses the following character class to strip punctuation, `[[:punct:]]`, which the [ruby doc](https://ruby-doc.org/core-2.4.1/Regexp.html) defines as:

> * `/[[:punct:]]/` - Punctuation character
>     

Not so helpful... As this is a POSIX character class, you can find many definitions online, none matching exactly what dev.to/ruby does. In the end, I used the same trick as the emojis (copying all possible punctuations in a heading, and looking at the generated anchor), and came up with:

```kotlin
val punctuationRegex = 
  Regex("[!\"#$%&'()*+,./:;<=>?@\\[\\]^_`{|}~՚Ꞌ′″‴〃-]")
```

The quotes were the most puzzling bit:

```xml
<a href="#-%CA%BB-%EA%9E%8C-%EA%9E%8C-%CB%AE">' ' ʻ ՚ Ꞌ ꞌ ′ ″ ‴ 〃 " ˮ</a>
```

Without the URL encoding, the anchor ends up: `-ʻ-ꞌ-ꞌ-ˮ` <sub>notice the two same characters in the anchor, even though they are different in the title? I guess it comes from the normalization they do on the content, which I don't have access to in Kotlin common (Java </sub> [<sub>Normalizer</sub>](https://docs.oracle.com/en/java/javase/11/docs/api/java.base/java/text/Normalizer.html)<sub>). Don't use special quotes in your titles, please 🙏.</sub>

After removing punctuation, the only thing left is to replace (consecutive) spaces with dashes.

```kotlin
title
  // sanitize and remove emojis ...
  .replace(punctuationRegex, "")
  .replace(Regex("\\s"), "-")
```
