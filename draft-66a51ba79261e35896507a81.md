### Forewords

The Twelve-Factor App is so relevant today but goes back to 2011. Initially drafted by Heroku engineers, it is a methodology (or set of best practices) for building *software-as-a-service*.

A software as a service is an app deployed remotely (e.g. in a cloud) and accessible through the internet (a web app). It can be written in any language and is stateless, in the sense that it relies on external services - called backing services - such as database or object storage to store its state.

In the modern area, most of what we develop can (should!) be categorized as software-as-a-service.

## The Twelve Factors

### I [Codebase](https://12factor.net/codebase)

> one codebase tracked in revision control, many deploys

Basically, keep your app in one central git repository (no one is still using subversion, right?). This same repository should be used (usable!) everywhere: prod, staging, locally.

This may seem obvious, but you can't imagine how many apps I see that are not even using version control! The setup necessary to run an app locally is also often lacking (which is why we provide the local environment for you at Divio).

## II. [Dependencies](https://12factor.net/dependencies)

> Explicitly declare and isolate dependencies

Use a package manager for dependencies, and explicitly list everything (ideally with pinning) your app needs to run. This also includes transitive dependencies!

This is now so easy (most programming languages offer tools for that), you don't have any excuse. Java → maven/gradle, Node → npm, Ruby → Bundler, etc.

Always pin your dependencies (except maybe for test packages), and don't forget the *transitive* dependencies! This will avoid surprises and crashes, and let you be sure of what is running at any point. It also helps with vulnerability scanning.

In Python, use something like Poetry (with a `poetry.lock`) or pipenv (with a `Pipfile.lock`). If you want to keep it simple, you can stay with a simple `requirements.txt`. But please, do not use `pip freeze > requirements.txt` ! Opt for a workflow such as `requirements.in` → [pip-compile](https://github.com/jazzband/pip-tools) → `requirements.txt` instead.

### III. [Config](https://12factor.net/config)

> Store config in the environment

Anything that can vary between deploys should be read from environment variables. This usually entails:

* connection information to backing services (database, cache, ...)
    
* credentials to external services (Facebook, AWS, kubectl, ...)
    
* hostname, size of the thread pool, timeouts, etc.
    

All of this should be strictly separated from the code. It doesn't mean you can have config in your code (e.g. the routing config in a web app)! Here config means anything you might want to change if you deploy somewhere else. If a config doesn't change, it may as well be a constant in the code.

From my experience, I would also add:

* be careful about default values. It is better to force every deployment to explicitly pass each configuration and crash early if one is missing, instead of trying to use a default value that may trigger an error later. Only provide defaults for configs related to performance and optimization (max memory, number of workers, size of a thread pool) - and even then, ask yourself if it is worth it.
    
* simplify local development by storing the configuration in one or more `.env` files. I usually use a `local/.env` committed to the repository, and another `local/.secret` uncommitted file for sensitive values or configs based on my machine. If you are not using Docker (which simplifies env as file), consider [direnv](https://direnv.net): it takes care of sourcing and outsourcing your env file every time you navigate in or out of your repository.
    
* use long, descriptive names. If your value has a unit, include it in the name. Between `TIMEOUT=4` and `CACHE_READ_TIMEOUT_SECONDS=5`, which do you prefer when debugging in production?
    
* Group related config using a prefix (not a suffix!). This way you can use `env | sort` or similar to get a better view of a deploy configuration.
    

### IV. [Backing services](https://12factor.net/backing-services)

> Treat backing services as attached resources.

A backing services is any service the app consumes over the network as part of its normal operation: databases, caching systems, queues, SMTP server, etc.

Treating backing service as a resource means they are "attached" to the app during deploy via config (e.g. `DATABASE_DSN=postgresql://user:password@db.example.com/db`). It doesn't matter to the app how the resource is managed or where it is running, as long as it can be accessed and complies to the expected protocol. This means you can swap a backing service for another (e.g. moving from an on-prem database to an AWS managed database) just by redeploying the app, and without any code change. There is also no distinction between internal and third-party services.

This pattern makes everything easier, both in production and locally (docker-compose.yml).

Some tips:

* design your app to be stateless. This means storage should also be treated as a backing service! Never work directly with the filesystem (except in rare, temporary cases, see VI), but consider object storage solutions (S3, MinIO) instead.
    
* Fonts and static assets should be baked in the image/executable, do not store them in media storages!
    
* do not make too much assumptions in the app code. For example, use well-known ORMs for databases and avoid version or vendor specific features unless you have a very good reason to do so. This way you can upgrade your backing service or switch implementations more easily.
    
* Make use of the strategy and bridge design patterns. Your app should be flexible and able to swiftly switch to the right "backend" implementation at runtime. Other design patterns (dependency-injection) are also handy!
    
* ease local development using a docker-compose and take advantage of mocking services such as [mailhog](https://github.com/mailhog/MailHog) (fake SMTP server). Don't forget to test the app with varying backing service implementations.
    

### V. [Build, release, run](https://12factor.net/build-release-run)

> Strictly separate build and run stages

The lifecycle of a codebase should follow 3 steps: build, release, and run, each with a decreasing level of complexity. A release in the Twelve-Factor App sense is a conjunction of a build + a config. Similarly, a run is not just a deploy but also happens e.g. during a restart after a crash.

[![](https://cdn.hashnode.com/res/hashnode/image/upload/v1722274240642/1842f0b6-f7a6-4dc7-bfa0-fda81e696d20.png align="center")](https://12factor.net/build-release-run)

In other words, whenever you deploy, you just make a release active. This may seem a bit obtuse, so let's take the Divio deployment process as an example.

At Divio, you have a codebase (a git repo), and settings for environment variables. When you deploy, what we do is:

1. build: we clone your repository and build a *build image* from your Dockerfile
    
2. release: we build a new *release image* that inherits from the build image and declares all the environment variables you defined in your settings (to make it clearer, we basically generate a Dockerfile containing only a `FROM build-image` and one or more `ENV XXX=YYY` directives and build it). In other words, your current environment variable settings are frozen within the release image.
    
3. run: we start containers based on the *release image*. There is nothing more to do at this step, since the release already contains all the configuration it needs.
    

We also treat releases as append-only ledgers: each release is clearly identifiable by a unique tag, never altered, and kept in an ordered list (each release has a previous release).

This process has many advantages. If a container crashes or needs to be moved to another host, we can start another without any other context information. Scaling is a piece of cake as well. A *rollback* can be done any time by starting containers based on the previous release image.

As you may have noticed, this means any configuration change must be released to become active. This differs a bit from the usual view of releases (just a tag or version of the codebase) - maybe the wording is not the best. Release artifact would be more accurate.

### VI. [Processes](https://12factor.net/processes)

> Execute the app as one or more stateless processes

Your app should consist of one or more processes that are all stateless and share-nothing. Any data that needs to persist must be stored in stateful backing services (→ IV. Backing services).

Let's say your app is a web server using e.g. gunicorn with python or express with NodeJS. In production, you will have multiple workers able to process requests in parallel. What the 12 factors tell you is to consider each request processing as strictly stateless: never assume anything cached on disk or in-memory will be available in another process or job. Each request is purely independent. Even more, a single process may be stopped due to a restart, maybe on another physical machine! if should be able to restart "from scratch" without errors.

Some notes:

* If your app is containerized, the only file system you can rely upon is the one baked into the image. Everything else is transcient and may be wiped out if not stored in a backing service.
    
* session state should always be stored externally, if possible using a service supporting time-exipration such as redis or Memcached.
    

### VII. [Port binding](https://12factor.net/port-binding)

> Export services via port binding

(This is something we are more and more used to thanks to containerization).

The contract with the execution environment is binding to a port to serve requests. This is not limited to web services (HTTP/HTTPS): nearly any kind of server software can be run via a process binding to a port and awaiting incoming requests.

Locally, you can visit the service directly (https://localhost:&lt;port&gt;/), while in production a *routing layer* takes care of mapping a public domain (and sometimes handling SSL termination) to the app's exposed port. The app itself should never handle the routing directly - it is the job of the execution environment.

Note that this port-binding approach means your app can easily be turned into a backing service for another app.

### VIII. [Concurrency](https://12factor.net/concurrency)

> Scale out via the process model

In the twelve-factor app, processes are a first class citizen. Instead of running multiple threads or processes within a single app instance, an app should be designed to handle multiple tasks simultaneously by scaling out separate, independent, unix-like processes. In other words, an app is a collection of one or more process types, each scalable on its own by increasing or decreasing the actual number of processes running.

Unclear? Let's take a web application. You will usually consist of a bunch of *web processes* able to handle the HTTP requests, and another set of *worker processes* to handle long-running tasks in the background. You could also have *clock processes* running scheduled jobs at regular interval, etc. If the app starts crawling under web requests, you can spawn an new web process. If the background tasks take to long to complete or spend to much time in the queue, add another worker process, etc.

If the other principles are respected, processes are stateless and independent. This means they can start and stop quickly and easily and span multiple physical machines. It makes the app highly scalable and reliable. It is also perfect for cloud environments, where resources can be dynamically allocated.

It also offers interesting cost savings and reliability. Instead of running everything on one big, expensive machine or VM, you can spread each worker type on multiple, low-budget VMs distributed in different zones or data centers.

### IX. [Disposability](https://12factor.net/disposability)

> Maximize robustness with fast startup and graceful shutdown

Since your app is composed of processes that can scale easily, you want to have very fast startup and shutdown times. This way you can really play with the number of processes and move them around as needed.

Even more important, it should be easy and safe to stop a process with a signal. Upon reception, the process should be able to quickly finish whatever it did and leave everything in a good state. For example, a terminating worker should stop processing new tasks, and either finish the current task or stop and return it to the queue so it can be picked by another worker.

Harder to achieve but as much important, your processes should be able to face sudden death (e.g. a hardware failure). Consider using proper transactions for databases, and robust queuing systems for background tasks able to return a task to the queue after a given timeout (or when a worker stops responding).

### X. [Dev/prod parity](https://12factor.net/dev-prod-parity)

> Keep development, staging, and production as similar as possible

Three kind of gaps may occur between development and production. The twelve factors strive to keep each as minimal as possible:

1. time gap: use continuous deployment and automated processes to be able to ship changes as fast as possible
    
2. personnel gap: break the silo between dev and ops (devops!). Developers who write the code should be involved in deploying and watching the production.
    
3. tools gap: it is tempting to use lightweight backing services in development and keep the heavy machinery for production (e.g. SQLIte vs PostgreSQL, or in-memory vs external Memcached). Resist the urge and ensure the app is tested as close as production as possible.
    

Adapters to different backing services are still useful, because they make porting to new backing services relatively painless. But all deploys of the app (developer environments, staging, production) should be using the same type and version of each of the backing services. For instance, use a docker-based redis locally and redislab a instance in production.
