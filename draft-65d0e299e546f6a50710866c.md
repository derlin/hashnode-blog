### Network namespace

See [(Baeldung) Docker network invisible.](https://www.baeldung.com/linux/docker-network-namespace-invisible)

Each container gets his own network namespace. It is usually visible on linux using `ip netns show`. However, docker doesn't create the namespaces under `/var/run/netns`, which is the place `ip` looks for. Instead, it creates it under `/proc/<container-pid>/ns`. To enter the network context of a container, run:

```bash
# Run a container
docker run --rm --name nginx -d nginx/latest
# Get the container's PID
pid=`docker inspect -f '{{.State.Pid}}' nginx`
# Launch a shell in the container's network namespace
nsenter --net=/proc/$pid/ns/net
# There, there is no network rule
iptables -L
Chain INPUT (policy ACCEPT)
target     prot opt source               destination

Chain FORWARD (policy ACCEPT)
target     prot opt source               destination

Chain OUTPUT (policy ACCEPT)
target     prot opt source               destination
```

## Publishing ports

(`http:alt` is `8080`)

When publishing a port, docker:

* creates two virtual ethernet devices (`veth)` and link them, one running the host namespace and one running on the container namespace. We can see it using `ip link show` on both the host and container netns. This is needed only once per container with exposed port(s).
    
* adds a NAT rule to iptables, to rewrite the destination of packets targeting `<host ip>:<published port>` to `<container ip>:<container port>` (one per port)
    
* adds a rule to the DOCKER chain allowing traffic to `<container ip>:<container port>`
    

When a packet arrives, it goes through the PREROUTING chain of the NAT. Then, the packet goes to INPUT, where the DOCKER chain kicks in.

And that's it!

```bash
# show details of docker0 bridge in pretty json format

```
