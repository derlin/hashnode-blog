Python is a great language, but its ecosystem is wild. Not as wild as Javascript/NodeJS, but still... PyPi has more than 576,000 packages (I know, I had to sync them all not long ago - it took me 3 days), 2,700 Python Enhancement Proposals (PEPs), and at least 4 major docstring conventions formats (NumPy, Google, reST, Epytext), etc.

The same is true for all the wiring around the code itself. It started with `requirements.txt`, but in the past years, lots of Python packaging and dependency management systems came to light. Poetry is the most well-known, but I should also mention Hatch, PDM (and conda for data science).

I was a very excited early adopter of Poetry. It is so powerful! You have nice commands, it does everything for you, what's not to like? I started using it everywhere on my side projects.

However, since I started working with Python professionally (with many projects and codebases), my opinions shifted (a lot).

The way I now manage Python dependencies (and packaging) may seem very "old-school" and "basic", but it is what makes the most sense to me now that I know better Python's internals and have to spend most of my time maintaining libraries and apps.

So let's talk about how I manage Python dependencies in 2024!

## Python Libraries

Python libraries are meant to be built into wheels and published in a PyPi index (most often https://pypi.org). Contrary to Apps, it is not just about dependencies: you also need a bunch of metadata and setup.

When setting up a project with poetry, it adds a `pyproject.toml` (good so far), but with lots of Poetry-specific shenanigans. More problematic, `poetry add` always pins the version of the package you are installing. At a library level, pinning dependencies is exactly what you should NOT do!

Let's take a step back. The roles of pinning are two-fold:

1. create reproducible builds, and
    
2. protect against breaking changes.
    

(1) doesn't make sense at a library level, because the build is not done by you but by your users. They should pin your library version, but the transitive dependencies you bring up should be loose enough to not conflict with others. If you start pinning your dependencies, you force your users' hand. Let's say you pin your package `A` to `3.2.1`. A user uses another library also relying on package `A`, but pinning it at version `3.2.2`. What happens? It's a straight conflict, both packages are now impossible to use together. Imagine now that package `A` is something like `requests` or `urllib` (used by the vast majority of libraries out there).

(2) only works with semantic versioning. Let's say you are using package `A` and you tested your library with version `3.2.1`. Pinning `A` to `^3` would give some leeway to users, while ensuring they do not suddenly use `4` that could introduce changes (as per the semver contract) that may break your library functionalities. Seems sensible, right?

If you agree, I will let you read this amazing article by Hynek, a major Python contributor:

%[https://hynek.me/articles/semver-will-not-save-you/] 

Semver is great in theory, but impossible in practice. Quoting Hyrum's law:

> "*With a sufficient number of users of an API, it does not matter what you promise in the contract: all observable behaviors of your system will be depended on by somebody*".

Whatever the *intent* of the maintainers, no minor or patch can guarantee non-breaking behaviors, and major changes may leave your library working as before.

Pinning your library dependencies \[to a major version\] will not shield you from an unintended change that introduces a bug to your users. Additionally, you risk annoying your users (badly) even though your library may be fine with the next major version.

So long story short, I now follow this simple process:

* I do not pin any dependency as a rule,
    
* I put a lot of effort into tests, trying to cover all major use cases and Python versions,
    
* I ensure those tests run as often as possible, usually using a daily or weekly schedule (e.g. scheduled workflows on GitHub or Gitlab)
    

This way, I can quickly react to any sudden change and either fix it or *temporarily* pin the problematic dependency until I have time to invest in the fix.

Now, that covers the dependency part. What about metadata, building, and publishing? Well, for pure Python libraries, I like to keep it simple. This currently means:

* a `pyproject.toml` for metadata and dependencies (and configuring other tools such as `ruff`), as introduced in PEP 621 and formalized in [PyPA's specification](https://packaging.python.org/en/latest/specifications/pyproject-toml/#pyproject-toml-spec),
    
* `setuptools` as the underlying build system, the `build` and `wheels` packages for building using the command-line, and `twine` for pushing to PyPi
    
* `setuptools-scm` for managing the version through git.
    

A good example is my mantelo library. Let's break down its `pyproject.toml` together. First, the build system:

```ini
[build-system]
build-backend = "setuptools.build_meta"
# Building needs the following installed
requires = [
  "setuptools>=61.0",
  "setuptools-scm[toml]>=6.2",
  "wheel",
]

[tool.setuptools]
platforms = ["any"]

[tool.setuptools.packages.find]
# Example: do not include tests in the wheel,
# but here you can customize how sources are discovered
exclude=['tests*']

[tool.setuptools_scm]
# Ask setuptools-scm to generate a file with the version
# read from git
write_to = "mantelo/version.py"
```

Then goes the project's metadata:

```ini
[project]
# metadata (used by PyPi)
dynamic = ["version"] # use setuptools_scm to read version from git
name = "mantelo"
description = "..."
license = { file = "LICENSE" }
authors = [{ name = "Lucy Linder", email = "lucy.derlin@gmail.com" }]
readme = "README.md"
keywords = ["..."]
classifiers = [
  "License :: OSI Approved :: Apache Software License",
  "Operating System :: OS Independent",
  "Programming Language :: Python",
  "...",
]
# Don't forget this one!
requires-python = ">=3.10"
# And, of course, unpinned dependencies
dependencies = [
  "attrs",
  "requests",
  "slumber",
]
```

To simplify the life of maintainers, I use optional dependencies, that people can install additionally using the `pip install .[dev,test]` syntax:

```ini
[project.optional-dependencies]
dev = [
  "build",
  "coverage",
  "tox",
  "...",
]

test = [
  "pytest",
  "...",
]
```

Finally, I tend to configure all the dev tools from `pyproject.toml` if they support it. For example:

```ini
[tool.ruff]
target-version = "py310"

[tool.pytest.ini_options]
markers = ["integration"]

[tool.mypy]
python_version = "3.10"
```

And that's it!

When I need to build, I can do:

```bash
pip install '.[dev]' # to install build
python build # will generate sdist + wheel in ./dist
twine check ./dist && twine upload ... # publish
```

BONUS:

* including a `Makefile` in the repo may help maintainers understand what is possible and how to e.g. run tests, lint the code, etc.
    
* `tox` , `ruff`, and `mypy` are to me just necessary in 2024. Ideally, they run within containers so every step of the development process is reproducible and easy to spin up on any machine whatsoever (and in CI!)
    
* CI should automate as much as possible: this includes releases and daily/weekly tests.
