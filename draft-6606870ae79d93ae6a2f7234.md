Mantelo is a super small yet super powerful library for interacting with the Keycloak Admin API. It handles authentication, querying, and only brings 3 dependencies.

More importantly, mantelo **guarantees support for all the endpoints of the Keycloak Admin API, even the beta and unreleased versions**. That is quite a bold claim, unmatched by the other libraries out there. How is this possible? You are in luck, this article reveals it all.

> Mantelo \[manˈtelo\], from German "*Mantel*", from Late Latin "*mantum*" means "*cloak*" in Esperanto.

## What is Keycloak

For those of you who never heard of [Keycloak](https://www.keycloak.org), it is a famous open-source identity and access management system (IDP). Coded in Java/Quarkus, it supports Single Sign-On (SSO), social login, user federation across multiple apps, and much more.

Aside from a very nice UI, Keycloak ships with a complete Admin REST API, allowing you to manage the different resources such as users, clients, and realms programmatically.

Keycloak is the IDP powering Divio, the awesome PaaS for which I work.

## Why mantelo

In Python, there are several Keycloak libraries available. I opted for the renowned [python-keycloak](https://python-keycloak.readthedocs.io/en/latest/). It worked well until I had to query for a resource that was not supported. I did my duty and submitted a [pull request](https://github.com/marcospereirampj/python-keycloak/pull/478) to add a `get_idp` method in August 2023. I had to wait three months for it to be merged, and a bit more for it to be released. I am not saying the maintainers were not competent, just that PRs in significant open-source projects take time.

Everything proceeded smoothly until I encountered the need to interact with the API for managing TOTP (Time-Based One-Time Password). Once again, I faced a wall: python-keycloak lacks support for any of the endpoints related to TOTP.

I had three choices:

1. Submit PRs and postpone everything for months (slow),
    
2. Use raw HTTP requests instead of python-keycloak in part of the code (ugly),
    
3. Create a library that features ALL the Admin endpoints, so I will never be in this conundrum again (fun!).
    

Guess which one I chose 😊.

## How to ensure all endpoints are always available? Is it even possible?

The Keycloak Admin REST API is already big and evolves on every release. How can a single maintainer (me) ensure mantelo consistently supports all endpoints and versions? Is this even possible?

YES. And the secret lies in not *implementing* all the endpoints, but *wrapping* them. Lucky me, this is exactly what slumber already does.

### The power of slumber

[slumber](https://slumber.readthedocs.io/en/latest/) is an amazing library (around for years) that deserves more recognition. From the README:

> Slumber is a Python library that provides a convenient yet powerful object-oriented interface to ReSTful APIs. It acts as a wrapper around the excellent [requests](http://python-requests.org/) [library](http://python-requests.org/) and abstracts away the handling of URLs, serialization, and request processing.

To better understand its power, let's use [dev.to's Forem API v1](https://developers.forem.com/api/v1) as an example:

```python
>> import slumber
>> api = slumber.API("https://dev.to/api")
# ⮕ GET https://dev.to/profile_images/{username}
>> api.profile_images("derlin").get()
{'type_of': 'profile_image', ...}
# ⮕ GET https://dev.to/api/articles?username=derlin&per_page=1
>> api.articles.get(username="derlin", per_page="1")
[{'type_of': 'article',
  'id': 1750725,
   ...
}]
# ⮕ POST https://dev.to/api/admin/users <body>
>> api.admin.users.post({"email": "a@a.com", "name": "a"})
HttpClientError: Client Error 401: https://dev.to/api/admin/users/
```

As you can see, the magic of slumber is to translate pythonic code to HTTP calls. The examples above showcase URL parameters, query parameters, and payload, but what about authentication? Well, slumber is just a wrapper around a [requests](https://requests.readthedocs.io/en/latest/) `Session`. The `auth` parameter passed to the `slumber.API` constructor maps directly to the [requests' auth](https://requests.readthedocs.io/en/latest/user/authentication/).

### Back to mantelo

To support all endpoints, always, mantelo thus builds upon slumber. The only big changes are:

1. mantelo provides its own authentication mechanism, which supports username/password authentication and client credentials. This seamlessly handles the fetching and refreshing of tokens for you.
    
2. mantelo overrides the way slumber computes the URLs to automatically convert underscores to dashes, as required by the Keycloak Admin API.
